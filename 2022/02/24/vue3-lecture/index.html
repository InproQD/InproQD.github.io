二、 逻辑复用方式的变化

在Vue2中，对于可复用的逻辑性代码抽离为单独的js文件，用mixins的方式引入

  • 抽离出来的逻辑性代码
// mixins/getAccount.js
export default {
  data(){
    return {
      hasAccount: false
    }
  },
  mounted(){
  const path = this.$route.path
  this.getAccount(path)
  }
  methods:{
   getAccount (route) {
      CustomerAuthApi.getAccount((result) => {
        if (tool.isEmpty(result)) {
          this.hasAccount = false
          this.$router.push('/sign-in')
          return
        }
        this.hasAccount = true
        this.$store.commit('setAccount', result)
      }, () => {
        this.$router.push('/sign-in')
      })
  },
}
  • 组件中引用复用代码
<template>
  <div></div>
</template>

<script setup lang="ts">
import getAccount from '@/mixins/getAccount.js'
export default {
  mixins:[getAccount],
}
</script>

在Vue3中，逻辑复用的代码文件命名以use开头

  • 复用性代码
// composables/useAccount.ts
import {ref, onmounted} from 'vue'
import {useRoute, useRouter} from 'vue-router'

const route = useRoute()
const router = useRouter()

export default function useAccount () {
  const hasAccount = ref(false)
  const path = route.path
  onMounted(()=>{
   getAccount(path)
  })
  
  const getAccount = (route) => {
      CustomerAuthApi.getAccount((result: any) => {
        if (tool.isEmpty(result)) {
          this.hasAccount = false
          router.push('/sign-in')
          return
        }
        hasAccount = true
      }, () => {
        router.push('/sign-in')
      })
  },
}
  • 引入useAccount
<template>
  <div></div>
</template>

<script setup lang="ts">
import useAccount from '@/composables/useAccount.ts'
const { hasAccount, path, getAccount } = useAccount()
onMounted(()=>{
  getAccount(path)
})
</script>

Vue2的mixins缺点就是，当在组件内引用多个mixins时，很难分清楚变量和方法的来源，并且多个mixins的生命周期会融合到一起运行，但是同名属性、同名方法无法融合，可能会导致冲突。而Vue3完美解决了这个问题，每一个复用代码的来源都很清晰，同名属性也不会混淆。

三、新的底层优化方式

01 新的diff算法和静态提升

  • 新的diff算法给每一个dom元素添加了可识别的标签，举个例子：
<template>
    <div id="content">
        <p class="text">Static</p>
        <p class="text">Static</p>
        <p class="text">{{ message }}</p>
        <p class="text">Static</p>
        ...
        <p class="text">Static</p>
    </div>
</template>

代码中的每一个Static都是静态节点，每一个Dom元素定义一个flag用来识别，值为-1，而message动态节点flag=1。flag为-1的节点会被缓存，当我们刷新页面的时候，静态节点将会从缓存中取出，动态节点（flage=1）通过diff算法重新渲染，这样节省了性能消耗。Vue3对于不参与更新的元素，会做静态提升。

02 Proxy和Object.defineProperty

  • Vue2响应式的基本原理是Object.defineProperty，这个方式存在一定的缺陷
watch: {
    'arrayA': {
     }
      deep: true,
      immediate: true
 }
 changeArray (arrayA) {
   arrayA[1] = 'change',
   arrayA.length = 12
 }

以上情况对数组的操作在Vue2中是监听不到的，pop、push、shift、unshift、splice、sort、reverse等方法操作数组可以监听到，或者通过Vue.$set()来改变数组

  • Vue 3是通过proxy直接代理整个对象来实现的，而不是像Object.defineProperty针对某个属性。所以，只需做一层代理就可以监听同级结构下的所有属性变化，包括新增属性和删除属性

四、构建优化

tree-shaking，也叫树摇，是Vue3组合式Api的优势之一，原理是筛选项目中无用的代码，以此来减少打包的体积，在Vue2项目中，我们只能通过webpack来配置tree-shaking。在vue2中，哪怕没有用到mounted等生命周期函数，在打包时也会包含这些基础函数。在示例代码中，我们只引入了onMounted这一个模块，在打包时，Vue3的tree-shaking特性不会让其他函数存在于基础包，只会有onMounted。

<template>
  <div>
  </div>
</template>

<script setup lang="ts">
import { onMounted } from 'vue'
</script>

除此之外，Vue3的API是基于函数形式，打包出来的体积会更小，举个例子

// utils
const tool = {
  method1() {},
  method2() {}
};
export default tool ;

//// 调用
import tool from '@/utils';
tool.method1();

以上代码通过webpack打包出来的结果为a={method1:function(){},method2:function(){}};a.method1();

// utils
export function method1() {}
export function method2() {}

// 调用
import { method1 } from '@/utils';
method1();

打包之后function a(){}a();。原因就是，当你引用一个对象时，无法对其属性进行打包筛选优化。而函数则可以更好的进行压缩，这也是Vue3函数式API的优势。

五、与Typescript的融合

Vue3是基于函数式的API，并且使用TypeScript开发的Vue3，对TypeScript具有天然的支持。

// setup和ts语法糖
<script setup lang="ts">
import { reactive, defineEmits, defineProps } from 'vue'
//defineProps会对其定义的参数进行类型推导
const props = defineProps({
  foo: { type: String, required: true },
  bar: Number
})

props.foo // string
props.bar // number | undefined
// ts 语法下函数传参必须要有类型
const function1 = (value: string) => {
  // 但是这里的value类型可选为string和undefined, 所以要加类型断言，告诉编译器，value确定为string类型
  value = props.foo as string
}
</script>

上述代码通过 defineProps定义了传值类型，因此，其他组件在引用该组件时并进行传值时，就能够自动推断出该传值的类型。如果引用方与被引用方的任何一个传值类型不一致，TS 就会抛错，让你可以提前规避很多错误。

六、特性优化

01 响应式API

  • ref译为reference, ref 有个「包装」的概念，它用来包装原始值类型，如 string 和 number ，我们都知道不是引用类型是无法追踪后续的变化的，ref 返回的是一个包含 .value 属性的对象。ref也可以声明对象或数组，自动用reactive进行包裹
<script setup lang="ts">

  const count = ref<number>(1);
  // 赋值
  count.value = 2;
  // 读取
  console.log('count.value :>> ', count.value);
</script>
// 但是在template中ref包装对象会被自动展开，不需要用.value
<template>  
  {{count}}
</template>
  • reactive是返回一个响应式的对象，和Vue2的data差不多，但因为Vue3的原理是proxy，所以可以对reactive进行更深层次的监听变化。注意：不能对reactive包裹的数据进行结果，会失去响应性。相比较于ref，reactive不需要.value来使用其内部的值。
<template>  
  <v-img url="portfolio.iconUrl"/>
</template>

<script setup lang="ts">
import { reactive } from 'vue'

const portfolio = reactive({
    iconUrl: '',
    logoUrl: ''
    })
</script>
  • toRefs的用途就是把响应式对象转化为普通对象，可以理解为ES6的结构，但是toRefs结构出来的数据并不失去响应性。
<script setup lang="ts">
import { reactive， toRefs } from 'vue'

const portfolio = reactive({
    iconUrl: '',
    logoUrl: ''
    })
    
const usePortfolio = toRefs(portfolio)

// usePortfolio 和 portfolio 已经建立了连接
portfolio.iconUrl = 'http://icon'
console.log(usePortfolio.iconUrl.value) // http://icon
//因为被解构出来的是对象，所以访问属性的值要用.value
</script>
  • toRef的用途跟toRefs相似，不过作用的目标是对象的某个属性，接收两个参数，第一参数是对象数据源的名称，第二参数是对象的key name，也就是属性名。
<script setup lang="ts">
import { reactive， toRefs } from 'vue'

const portfolio = reactive({
    iconUrl: '',
    logoUrl: ''
    })
    
const useLogoUrl = toRef(portfolio， 'logoUrl')

// 更改该 ref 会更新源属性
useLogoUrl.value= 'http://logo'
console.log(portfolio.logoUrl) // http://logo

// 注意，以下代码不起效，因为ref接收的是一个纯数值
const useLogoUrl = ref(portfolio.logoUrl)

</script>

在这里对toRef和toRefs进行解释，这两个的原理就是浅拷贝，解决的问题就是，结构响应式对象后，新的对象并不失去响应性。实用的场景：当我们引入复用的代码逻辑时，不需要重复声明响应式数据，用toRef和toRefs对源数据进行结构即可。

02 v-model优化

  • 在Vue2场景中，父子组件传值通过v-model进行传递，但是却无法使用多个v-model，只能配合.sync来实现
<template>
 <wap-form
      :row="true"
      v-model="layouts.data.formData"
      :layout-setting.sync="layouts.settings.personal"
    ></wap-form>
</template>

如代码中展示的那样，Vue2只支持一个v-model语法糖，其他的值传递就需要用:属性值

  • 在Vue3中针对这种情况做了一个整合，多个需要传递的数据都可使用v-model语法糖
<template>
    <text-field-box
          v-model="data.value"
          v-model:label="data.label"
          v-model:variant="data.variant"
          v-model:icon="data.icon"
        >
    </text-field-box>
</template>
<script setup lang="ts">
import { reactive } from 'vue'
import TextFieldBox from '@/components/base/TextFieldBox.vue'

const data = reactive({
  value: '',
  label: '',
  variant: '',
  icon: ''
})
</script>

我们在上面用了四个v-model 语法糖

v-model语法糖 对应子组件的prop名 对应的emit事件名
v-model modelValue update:modelValue
v-model:label label update:label
v-model:variant variant update:variant
v-model:icon icon update:icon

03 全局API

  • createApp API返回一个app实例，在这个实例上挂载全局用到的方法和属性
import { createApp } from 'vue'
const app = createApp(App)
  .use(vuetify)
  .use(router)
  .use(store)
  
app.mount('#app')
  • app.config.globalProperties用来定义全局变量，是一个所有组件实例访问到的全局属性的对象。在Vue2中我们使用的Vue.prototype已经被替代
// main.ts文件中定义全局变量
app.config.globalProperties.$message = 'Feng Huang'

// 任意组件使用该变量
<script setup lang="ts">
import { onMounted } from 'vue'

onMounted(()=>{
  console.log(this.message ) // Feng Huang
})
</script>
  • app.directive自定义指令，一般在单独的ts文件中定义，并在mani.ts中引入
// directive.ts
const app = createApp(App);
 
app.directive('page-behaviour',{
    beforeMount(element: any, binding: any, node: any) {
    if (Object.prototype.hasOwnProperty.call(binding, 'value')) {
      let path = binding.value.path
      if (path.indexOf('/welcome') !== -1) {
        path = '/welcome'
      }
      CommonRequest.addCustomerBehaviorRecords(path, DataType.EventTypeEnum.ENTER_PAGE.value, binding.value.eventSource)
    }
  },
  beforeUnmount(element: any, binding: any, node: any) {
    if (Object.prototype.hasOwnProperty.call(binding, 'value')) {
      let path = binding.value.path
      if (path.indexOf('/welcome') !== -1) {
        path = '/welcome'
      }
      CommonRequest.addCustomerBehaviorRecords(path, DataType.EventTypeEnum.LEAVE_PAGE.value, binding.value.eventSource)
    }
  }
})

04 emits变化

  • emits一般作为子组件给父组件传值的一个途径，在Vue2中以this.$emit来使用，在Vue3中通过defineEmits函数来声明，具体用法如下
// 子组件内
<script setup lang="ts">
// defineEmits 只能在<script setup> 语法糖中使用, 可直接使用，不用import

// 接收数组为参数，数组中可定义多个事件名
const emits = defineEmits(['change', 'destroy'])
onMounted(() => {
  // 第一参数为抛出的事件名，第二参数为携带的值
  emits('change', 111)
})
</script>

// 父组件接收
<template>
      <children
      @change="handleChange"
      >
      </children> 
</template>
<script setup>
const handleChange = (param) => {
  console.log(param) // 111
}
</script>
  • 在之前的v-model的语法糖中，可配合emits进行传值，如下所示：
// 父组件
<template>
    <text-field-box
          v-model="data.value"
          v-model:icon="data.icon"
        >
    </text-field-box>
</template>
<script setup lang="ts">
import { reactive } from 'vue'
import TextFieldBox from '@/components/base/TextFieldBox.vue'

const data = reactive({
  value: '',
  icon: ''
})
</script>


// 子组件内
<template>
    <v-text-field
      @input="$emit('update:value', $event.target.value)"
      @keydown.ctrl="$emit('update:icon', 'email')"
    >
    </v-text-field>
</template>
// 当按下键盘上的ctrl键，就会变化子组件上的icon，update:icon对应v-model:icon，传的email赋给父组件的data.icon

Vite工具

对比Webpack

Vite初始化项目会比其他构建工具快很多，一般也就在5秒之内便能初始化项目，但是第一次打开网页的时间较长，因为在这个阶段加载js和cs包，需要等待一段时间。意思是，vite将部分打包的程序放在浏览器来进行，这也是问什么Vite初始化项目极快的原因。但是之后更待代码对页面进行更新就会比第一次快很多，因为浏览器将第一次转换的内容缓存起来。

Pinia插件

与VUEX的对比

  • 比Vuex更轻量
  • 天然支持TypeScript
  • Vuex中的mutations在Pinia中已经不存在，整合为actions
  • Pinia调用actions的方法更简便，store.function即可调用，不同于Vuex的$dispatch

    使用案例

  • 文件结构
  • index.ts
import { createPinia } from 'pinia'
// 每个module单独引入
import { useAuthStore } from '@/modules/auth'
import { useLoanStore } from '@/store/modules/loan'
import { useNotificationStore } from '@/store/modules/notification'
import { usePortfolioStore } from '@/store/modules/portfolio'
import { useRepairStore } from '@/store/modules/repair'
import { useWorkflowStore } from '@/store/modules/workflow'
// 持久化插件
import piniaPluginPersistedstate from 'pinia-plugin-persistedstate'

const store = createPinia()
store.use(piniaPluginPersistedstate)

export { useAuthStore, useLoanStore, useNotificationStore, usePortfolioStore, useRepairStore, useWorkflowStore }

export default store
  • 以authStore为例
import { defineStore } from 'pinia'
import { storePersistence } from '@/utils/storePersistence'
export const useAuthStore = defineStore('auth', {
  state: () => {
    return {
      account: {} as any,
      identity: {},
      profilePage: null
    }
  },
  getters: {},
  actions: {
  async getAccount() {
      const notificationStore = useNotificationStore()
      return await new Promise((resolve, reject) => {
        void CustomerAuthApi.getAccount(
          (result: any) => {
            this.account = result
            notificationStore.initWebSocket(result)
            notificationStore.getSiteMessage()
            resolve('success')
          },
          () => {
            reject(new Error())
          }
        )
      })
    }
  },
  // 持久化选项
  persist: true
})
  • 在组件中使用
<script setup lang="ts">
import { onMounted } from 'vue'
import { useAuthStore } from '@/store'

const authStore = useAuthStore()

onMounted(()=>{
  //调用actions的方法
  authStore.getAccount()
  // 调用state中的属性值
  consol.log(authStore.$state.account)
  // 修改state的属性值
  authStore.$patch((state) => {
    state.account = {}
  })
})

</script>
